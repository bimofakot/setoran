## [2026-04-10] Real-time Listener (onSnapshot)

### Sebelumnya
- `useTransactions` dan `useCategories` menggunakan `getDocs` (one-shot fetch).
- Setiap mutasi (add/update/delete) memanggil ulang `fetchTransactions()` secara manual.
- Tidak ada sinkronisasi otomatis antar tab atau setelah perubahan dari luar.

### Sesudahnya
- Keduanya menggunakan `onSnapshot` yang di-setup di dalam `useEffect` dengan `onAuthStateChanged` sebagai guard.
- Listener dibersihkan (`unsubscribe`) saat user logout atau komponen unmount — tidak ada memory leak.
- `fetchTransactions` dipertahankan sebagai no-op untuk backward compatibility.
- Mutasi (add/update/delete) tidak perlu memanggil fetch ulang — Firestore listener otomatis memperbarui state.

### Pola Cleanup
```ts
useEffect(() => {
  let unsubSnap: (() => void) | null = null;
  const unsubAuth = onAuthStateChanged(auth, (user) => {
    if (unsubSnap) { unsubSnap(); unsubSnap = null; }
    if (!user) return;
    unsubSnap = onSnapshot(query(...), (snap) => { /* update state */ });
  });
  return () => { unsubAuth(); if (unsubSnap) unsubSnap(); };
}, []);
```

## [2026-04-10] Smart Comparison Logic

### Sebelumnya
- `Analytics.tsx` selalu membandingkan bulan ini vs bulan lalu, hardcoded.
- Tidak mengikuti periode aktif yang dipilih user.

### Sesudahnya
- Comparison bersifat **dinamis** mengikuti `mode` aktif:
  - `today` → bandingkan dengan kemarin (`offset - 1`)
  - `week` → bandingkan dengan minggu lalu
  - `month` → bandingkan dengan bulan lalu
  - `year` → bandingkan dengan tahun lalu
  - `custom` → perbandingan dinonaktifkan (tidak ada referensi periode sebelumnya)
- Label otomatis menyesuaikan: "vs kemarin", "vs minggu lalu", dst.
- Tombol **"Tampilkan Semua"** merangkum perbandingan harian + mingguan + bulanan sekaligus dalam satu view.

## [2026-04-10] Responsive Breakpoints — Standar Baru

### Breakpoint yang Digunakan

| Nama     | Nilai     | Tailwind | Keterangan                              |
|----------|-----------|----------|-----------------------------------------|
| mobile   | < 640px   | (default)| 1 kolom, padding minimal, font lebih kecil |
| sm       | ≥ 640px   | `sm:`    | Padding normal, font normal             |
| md       | ≥ 768px   | `md:`    | Sidebar muncul, bottom nav hilang       |
| lg       | ≥ 1024px  | `lg:`    | Grid 2 kolom untuk Analytics & Kategori |

### Aturan Grid

- **Stat cards (Summary):** `grid-cols-1 sm:grid-cols-3` — 1 kolom di mobile, 3 kolom mulai sm
- **QuickStats:** `grid-cols-2 lg:grid-cols-4` — 2 kolom di mobile/tablet, 4 kolom di desktop
- **Analytics bottom (Pie + Comparison):** `grid-cols-1 lg:grid-cols-2` — stack di bawah lg
- **Kategori (ProfilePage):** `grid-cols-1 lg:grid-cols-2` — stack di bawah lg

### Overflow Handling

- **Filter pills:** Menggunakan class `.pill-row` (CSS custom) — `overflow-x: auto`, `scrollbar-width: none`, `flex-shrink: 0` pada setiap pill. User bisa scroll horizontal tanpa teks terpotong.
- **Main content:** `overflow-x: hidden` + `min-width: 0` mencegah flex child meluber keluar container.
- **Cards:** Semua `.card`, `.card-sm`, `.stat-card` mendapat `min-width: 0`.

### Padding Dinamis

- Main content: `px-3 sm:px-5 lg:px-8` / `py-5 sm:py-7`
- Card: `padding: 1.25rem` default, `1.5rem` mulai sm (via `@media`)
- Stat card: `padding: 1.125rem 1.25rem` default, `1.5rem` mulai sm

### Floating Window

- Desktop: `left: calc(var(--sidebar-w) + 1rem)`, `width: min(320px, calc(100vw - sidebar - 2rem))`
- Mobile (< 768px): `left: 0.75rem; right: 0.75rem; width: auto`

### Sidebar Mobile

- `position: fixed; top: 0; left: 0; z-index: 50` — selalu di atas semua konten
- Overlay backdrop di `z-40`, sidebar di `z-50`



### Masalah
`initUserCategories` sebelumnya selalu berjalan saat login (`onAuthStateChanged`) dan menambahkan kembali kategori default yang sudah dihapus user. Ini membuat user tidak bisa benar-benar menghapus kategori.

### Solusi
```ts
// userSetup.ts
export const initUserCategories = async (userId: string) => {
  const snapshot = await getDocs(categoriesRef);
  if (!snapshot.empty) return; // ← kunci: hanya seed jika koleksi kosong
  await Promise.all(DEFAULT_CATEGORIES.map(...));
};
```

**Prinsip:** Seed data awal hanya sebagai *bootstrap* saat user baru pertama kali mendaftar. Setelah itu, data sepenuhnya milik user — tidak ada kategori yang dikunci atau dipaksa kembali.

**Pengaman:** Jika user menghapus semua kategori, koleksi menjadi kosong, dan seed akan berjalan kembali saat login berikutnya — memberikan titik awal yang bersih tanpa kehilangan kontrol.

## [2026-04-10] Unified Period Filter di Analytics

### Sebelumnya
- `Analytics` menerima `transactions` yang sudah difilter dari Dashboard.
- Tidak ada kontrol periode di halaman Analisis.
- Filter Custom sudah dihapus dari `DateRangeFilter` (diganti `PeriodNavigator`) tapi belum ada di Analytics.

### Sesudahnya
- `Analytics` memiliki `PeriodNavigator` sendiri dengan props opsional untuk controlled mode dari Dashboard.
- Jika props `mode/offset/customRange` diberikan dari Dashboard, Analytics menggunakannya (sinkron).
- Jika tidak, Analytics mengelola state periodenya sendiri (standalone).
- Filter **Custom** dikembalikan: date picker `<input type="date">` muncul saat pill "Custom" diklik.
- `Analytics` memfilter `transactions` (semua data) secara internal — tidak bergantung pada `filteredTransactions` dari Dashboard.

## PROSEDUR TETAP (Standing Protocol)

### P-1: Multi-Language Audit
Setiap kali ada perubahan fitur atau dokumentasi:
1. Cek apakah ada `README.id.md` (atau file `.id.` lainnya) di root folder.
2. **Wajib update keduanya secara sinkron** — jangan biarkan satu versi tertinggal.
3. Jangan hapus seluruh isi file — edit bagian yang relevan dan tambahkan fitur baru.
4. Jika ada bagian yang dihapus, tambahkan catatan changelog di dalam file.

### P-2: Descriptive Git Commit & Push
Setiap kali tugas selesai atau protokol diperbarui:
1. `git add -A`
2. Commit dengan format **Conventional Commits**:

| Tipe | Kapan digunakan |
|---|---|
| `feat:` | Fitur baru |
| `fix:` | Perbaikan bug |
| `refactor:` | Pembersihan kode tanpa mengubah fitur |
| `docs:` | Perubahan dokumentasi saja |
| `style:` | Perubahan CSS/styling tanpa logika |
| `chore:` | Tugas rutin/config (`.gitignore`, protokol, deps) |
| `perf:` | Optimasi performa dan kecepatan |
| `build:` | Perubahan sistem build atau dependensi |

3. `git push`
4. Contoh baik: `feat: smart comparison, real-time onSnapshot, responsive breakpoints`
5. Jangan gunakan pesan generik seperti "update" atau "fix stuff".

### P-3: Smart .gitignore Audit
Setiap kali ada file/folder baru ditambahkan ke proyek:
1. Periksa apakah file tersebut mengandung secret, credential, atau data privat.
2. Kategori yang **wajib masuk** `.gitignore`:
   - `*.env`, `.env.local`, `.env.*.local`, `.env.production` — variabel lingkungan
   - `serviceAccount*.json`, `*credentials*.json`, `*secret*.json` — service account Firebase/GCP
   - `.firebase/` — cache deploy Firebase
   - `firestore.debug.log`, `firebase-debug.log` — log debug Firebase
   - `*.log`, `logs/` — semua log aplikasi
   - `.cache/`, `*.tmp`, `*.temp` — file sementara
   - `.DS_Store`, `Thumbs.db`, `desktop.ini` — file sistem OS
3. File yang **boleh di-track** meski terkait Firebase:
   - `.firebaserc` — hanya berisi project alias, tidak ada secret
   - `firebase.json` — konfigurasi hosting, tidak ada secret
   - `.env.example` — template tanpa nilai nyata
4. Jalankan `git status` sebelum commit untuk memastikan tidak ada file sensitif yang ter-stage.

## [2026-04-10] Auto-hide Bottom Nav

### Logika
```ts
useEffect(() => {
  const onScroll = () => {
    const y = window.scrollY;
    setNavHidden(y > lastScrollY.current && y > 80); // hide saat scroll down > 80px
    lastScrollY.current = y;
  };
  window.addEventListener('scroll', onScroll, { passive: true });
  return () => window.removeEventListener('scroll', onScroll);
}, []);
```
- Scroll **ke bawah** → `navHidden = true` → class `hidden-nav` → `transform: translateY(100%)`
- Scroll **ke atas** → `navHidden = false` → nav muncul kembali
- Threshold 80px mencegah flicker saat tap/micro-scroll
- Transisi: `cubic-bezier(0.4,0,0.2,1)` 300ms — halus dan natural
- Hanya aktif di mobile (`md:hidden`)

## [2026-04-10] Date Persistence Fix (WIB)

### Masalah
`new Date('2026-04-10')` di JavaScript mem-parse string `YYYY-MM-DD` sebagai **UTC midnight**, bukan lokal. Di timezone WIB (UTC+7), ini menjadi `2026-04-09T17:00:00` — satu hari mundur.

### Solusi
```ts
// Parse manual untuk hindari UTC shift
const [y, m, d] = dateStr.split('-').map(Number);
const combinedDateTime = new Date(y, m - 1, d, now.getHours(), ...);
```
- `new Date(y, m-1, d, ...)` selalu menggunakan **local timezone** — tidak ada UTC shift
- `todayWIB()` di TransactionForm: offset manual `+7*60*60*1000` untuk default date yang akurat
- `Timestamp.now()` untuk `createdAt`/`updatedAt` sudah menggunakan server time Firebase — tidak perlu diubah
