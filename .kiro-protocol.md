## [CORE-1] Anti-White Policy

Aplikasi ini **tidak menggunakan pure white** (`#ffffff`) di mana pun.
Semua warna menggunakan CSS variables yang terdefinisi di `:root` (dark) dan `[data-theme="light"]`.

### Dark Theme (default)
| Variable | Nilai | Keterangan |
|---|---|---|
| `--bg-base` | `#05080f` | Background halaman |
| `--bg-surface` | `#090e1a` | Background sidebar/card |
| `--bg-elevated` | `#0e1525` | Background tooltip/elevated |
| `--accent` | `#7c3aed` | Violet utama |
| `--accent-light` | `#a78bfa` | Violet terang (teks) |
| `--green` | `#10b981` | Warna pemasukan |
| `--red` | `#ef4444` | Warna pengeluaran |
| `--text-primary` | `#f0f4ff` | Teks utama |

### Light Theme — Pale Lavender Clay
| Variable | Nilai | Keterangan |
|---|---|---|
| `--bg-base` | `#ede9f4` | Background halaman |
| `--bg-surface` | `#e4dff0` | Background sidebar/card |
| `--accent` | `#6d28d9` | Violet utama |

**Aturan:** Jangan hardcode `#fff`, `white`, atau `bg-white`. Selalu gunakan `var(--bg-*)`.

## [CORE-2] Real-time Sync (onSnapshot)

`useTransactions` dan `useCategories` menggunakan `onSnapshot` Firestore, bukan `getDocs`.

### Pola Cleanup Wajib
```ts
useEffect(() => {
  let unsubSnap: (() => void) | null = null;
  const unsubAuth = onAuthStateChanged(auth, (user) => {
    if (unsubSnap) { unsubSnap(); unsubSnap = null; }
    if (!user) return;
    unsubSnap = onSnapshot(query(...), (snap) => { /* update state */ });
  });
  return () => { unsubAuth(); if (unsubSnap) unsubSnap(); };
}, []);
```

**Aturan:** Listener wajib dibersihkan saat logout dan unmount. Tidak boleh ada `getDocs` untuk data yang perlu real-time.

## [CORE-3] Category Locking — "Lainnya" System Category

Kategori dengan nama `Lainnya`, `Lainnya (Pemasukan)`, dan `Lainnya (Pengeluaran)` adalah **kategori sistem**.
- Tombol hapus disembunyikan (diganti badge "sistem")
- Tidak bisa dihapus via `handleDeleteCategory`
- Wajib selalu ada sebagai fallback kategori bebas

**Aturan:** Jangan pernah menghapus guard ini dari `ProfilePage.tsx`.

## [CORE-4] Autoseed — Bootstrap Only, No Force

```ts
export const initUserCategories = async (userId: string) => {
  const snapshot = await getDocs(categoriesRef);
  if (!snapshot.empty) {
    // cek duplikat nama|type sebelum tambah
    const existing = new Set(docs.map(d => `${d.name}|${d.type}`));
    await Promise.all(missing.map(cat => setDoc(...)));
    return;
  }
  // koleksi kosong → seed penuh
};
```

**Prinsip:** Seed adalah *bootstrap* — hanya berjalan saat koleksi kosong atau ada kategori default yang hilang. Data milik user sepenuhnya setelah itu.

## P-1: Multi-Language Audit
Setiap kali ada perubahan fitur atau dokumentasi:
1. Cek apakah ada `README.id.md` (atau file `.id.` lainnya) di root folder.
2. **Wajib update keduanya secara sinkron** — jangan biarkan satu versi tertinggal.
3. Jangan hapus seluruh isi file — edit bagian yang relevan dan tambahkan fitur baru.
4. Jika ada bagian yang dihapus, tambahkan catatan changelog di dalam file.

---

## P-2: Descriptive Git Commit & Push
Setiap kali tugas selesai atau protokol diperbarui:
1. `git add -A`
2. Commit dengan format **Conventional Commits**:

| Tipe | Kapan digunakan |
|---|---|
| `feat:` | Fitur baru |
| `fix:` | Perbaikan bug |
| `refactor:` | Pembersihan kode tanpa mengubah fitur |
| `docs:` | Perubahan dokumentasi saja |
| `style:` | Perubahan CSS/styling tanpa logika |
| `chore:` | Tugas rutin/config (`.gitignore`, protokol, deps) |
| `perf:` | Optimasi performa dan kecepatan |
| `build:` | Perubahan sistem build atau dependensi |

3. `git push`
4. Jangan gunakan pesan generik seperti "update" atau "fix stuff".

---

## P-3: Smart .gitignore Audit
Setiap kali ada file/folder baru ditambahkan ke proyek:
1. Periksa apakah file tersebut mengandung secret, credential, atau data privat.
2. Kategori yang **wajib masuk** `.gitignore`:
   - `*.env`, `.env.local`, `.env.*.local`, `.env.production`
   - `serviceAccount*.json`, `*credentials*.json`, `*secret*.json`
   - `.firebase/`, `firestore.debug.log`, `firebase-debug.log`
   - `*.log`, `logs/`, `.cache/`, `*.tmp`, `*.temp`
   - `.DS_Store`, `Thumbs.db`, `desktop.ini`
3. File yang **boleh di-track**: `.firebaserc`, `firebase.json`, `.env.example`
4. Jalankan `git status` sebelum commit untuk memastikan tidak ada file sensitif yang ter-stage.

---

## P-4: Code Cleanup Protocol
Sebelum setiap rilis atau setelah sprint besar:
1. Jalankan `npx tsc --noEmit --noUnusedLocals --noUnusedParameters` — pastikan 0 error.
2. Cari dan hapus:
   - `console.log` / `console.debug` yang bukan error handler
   - Fungsi yang diekspor tapi tidak diimport di mana pun
   - File stub kosong (seperti `// replaced by X`)
   - File template sisa (seperti `App.css` dari Vite scaffold)
   - Dokumen markdown redundan yang isinya sudah ada di README
3. **Jangan hapus** `console.error` di catch block — itu error handler yang valid.
4. **Jangan hapus** no-op backward-compat stubs yang masih ada di return hook.
5. Build wajib hijau (`npm run build`) sebelum commit cleanup.

---

## P-5: Auto-Maintenance — Sticky Nav Integrity
Setiap kali menambah fitur navigasi atau layout baru:
1. **Topbar wajib `position: fixed; top: 0; z-index: 35`** — tidak boleh ikut scroll.
2. Kompensasi padding-top wajib ada di `.main-content` untuk mobile (`padding-top: 3.75rem`).
3. Cek konflik `overflow: hidden` pada parent — property ini mematikan `position: sticky`. Gunakan `fixed` sebagai gantinya jika parent punya overflow.
4. Z-index hierarchy yang wajib dijaga:
   - Topbar: `z-35`
   - Bottom nav: `z-40`
   - Sidebar desktop: `z-40`
   - Sidebar mobile overlay: `z-50`
   - Floating windows: `z-50`
5. Setelah perubahan layout, verifikasi: scroll ke bawah → topbar tetap diam, bottom nav slide away.

---

## P-6: Category Integrity Protocol
Setiap kali ada operasi massal pada kategori:
1. Kategori sistem (`Lainnya`, `Lainnya (Pemasukan)`, `Lainnya (Pengeluaran)`) **tidak boleh dihapus** — tampilkan badge "sistem", sembunyikan tombol hapus.
2. Deduplicate menggunakan key `nama|type` — keep first occurrence, delete sisanya via `writeBatch`.
3. Transaksi menyimpan **nama kategori sebagai string** (bukan ID) — re-link otomatis tidak diperlukan saat dedupe nama yang sama.
4. Tombol "Reset Duplikat" tersedia di halaman Kelola Kategori dengan konfirmasi sebelum eksekusi.
