## 🧠 LESSONS LEARNED (DO NOT REPEAT)
- **Fix #9 — Multi-page Layout Balancing:** Estimate total height of all post-table elements (entry count + summary box + divider + signature + disclaimer ≈ 113 units). If `finalY + POST_TABLE_HEIGHT > pageH - marginBottom`, call `doc.addPage()` before drawing any post-table block. This prevents orphaned signature areas on empty pages. Signature area must have at least 40 units of vertical gap below the summary box.
- **Fix #10 — Consistent Branding Policy:** App display name is **Keuanganku** — used in PDF header, report title, meta section, WhatsApp messages, and copy text. The name **Setoranku** is reserved exclusively for URL/domain references and footer validation text (as platform provider). Never use "Setoranku" or "SETORANKU" as a visible app title inside the UI or documents.
- **Fix #11 — Page Numbering (Two-pass):** jsPDF does not know total page count until all pages are rendered. Strategy: render footer with placeholder text, then loop `doc.setPage(i)` after full render, cover placeholder with white rect, and overwrite with final `Halaman X dari Y` string.
- **Fix #12 — Signature Alignment:** Signature block (location, line, name) must use `align: 'center'` with `sigX` calculated as the center of the right half of the page: `pageW / 2 + (pageW / 2 - 14) / 2`. Do NOT use a fixed offset like `pageW - 70`.
- **Fix #14 — PDF DateTime with Weekday:** Use `weekday: 'long'` in `toLocaleString`/`toLocaleDateString` options to include nama hari (e.g. "Kamis, 9 April 2026"). Split meta section into two rows: `Tanggal` and `Pukul` for clean alignment — never combine date+time in one `fmtDateTime` call for the meta header.
- **Fix #15 — Single Footer with Wrap:** Merge disclaimer + branding into one `FOOTER_FULL` string. Use `doc.splitTextToSize(FOOTER_FULL, pageW - 28)` so text wraps instead of being clipped. Adjust `startY` of footer text upward by `(lines.length - 1) * 3.5` to keep page number at the bottom edge.
- **Fix #16 — Table No. Column:** Add auto-numbered `No.` column as index 0 with `cellWidth: 10, halign: 'center'`. Columns `No.`, `Tanggal & Jam`, and `Deskripsi` all use `halign: 'center'`. Empty description renders as `'-'` centered. Shift `startY` to 50 (was 44) to accommodate 3-row meta section.
- **Fix #17 — Signature on Last Page Only:** Signature block is drawn only after `lastAutoTable.finalY`, never inside `didDrawPage`. The `POST_TABLE_HEIGHT` guard uses `pageH - MARGIN_BOTTOM - 6` to account for the taller footer area.
- **Fix #18 — Edit Transaction:** `updateTransaction` was already in `useTransactions` and `TransactionForm` already supported `initialData` + "Edit Transaksi" title. Only Dashboard wiring was missing. Pattern: single `editingTransaction` state (type `Transaction | undefined`); `handleAddTransaction` branches on its presence — calls `updateTransaction` if set, `addTransaction` if not. Reset `editingTransaction` to `undefined` on form close. No changes to hook or form component.
- **Fix #19 — Dynamic Category CRUD:** Added `addCategory` and `removeCategory` to `useCategories` hook. Both call Firestore then reload via shared `load()` callback (wrapped in `useCallback`). UI in `ProfilePage`: type toggle (Pemasukan/Pengeluaran) + name input + add button; category lists split by type with hover-reveal delete button. Default categories seeded by `initUserCategories` on signup remain deletable by user.
- **Fix #20 — Filter Sync (Custom Range Persistence):** Extracted `customRange` state `{ start, end } | null` in Dashboard. `useEffect` for filtering now branches: if `selectedRange === 'custom' && customRange` use stored dates, else use `getDateRange()`. This ensures custom filter re-applies correctly when new transactions arrive. `Analytics` now receives `filteredTransactions` (not raw `transactions`) so charts always reflect the active period filter.

## 🔒 DATA ISOLATION STANDARD (PERMANENT RULE)
- **Path-based isolation is the primary mechanism.** All user data lives under `users/{uid}/...`. Never store user financial data at a top-level collection (e.g., `transactions/{id}`).
- **No cross-user query filter needed.** Because isolation is enforced at the Firestore path level, queries within `users/{uid}/transactions` or `users/{uid}/categories` are already scoped — no `.where('userId', '==', uid)` required on subcollections.
- **Exception — username lookup:** `isUsernameAvailable` and `getEmailByUsername` in `userSetup.ts` query the top-level `users` collection to resolve username → email during login/register. This is intentional and safe — only `email` and `username` fields are read, never financial data.
- **Firestore Security Rules (enforced, v2):**
  ```
  match /users/{userId} {
    allow read: if true;  // untuk username lookup saat login/register
    allow write: if request.auth.uid == userId;
    match /transactions/{transactionId} {
      allow read, write: if request.auth.uid == userId;
    }
    match /categories/{categoryId} {
      allow read, write: if request.auth.uid == userId;
    }
  }
  ```
- **Alasan `allow read: if true` pada `/users/{userId}`:** `isUsernameAvailable` dan `getEmailByUsername` perlu query koleksi `users` sebelum user terautentikasi (saat registrasi/login). Hanya field `username` dan `email` yang terbaca — tidak ada data finansial di dokumen root user.
- **New features must follow:** Any new collection or document must be placed under `users/{uid}/` and covered by the wildcard rule above. Never create a shared/global collection accessible by multiple users.
- **Design Decision:** Analytics placed in a separate "Analisis" tab (not inline with dashboard) to avoid layout overload on mobile. Tab bar appears below the main header, hidden when Profile view is active.
- **No External Chart Library:** All charts built with native SVG + Tailwind. Zero new dependencies. Keeps bundle size minimal.
- **Charts Implemented:**
  1. **Trend Line Chart** — Income vs Expense for last 7 days. SVG polyline with dot markers and date labels. Y-axis normalized to `maxDaily` value.
  2. **Category Pie Chart** — Top 5 expense categories. SVG arc paths computed from cumulative angles. Color palette: blue, purple, pink, amber, emerald.
  3. **Monthly Comparison Bar** — Side-by-side horizontal bars for this month vs last month (income + expense). Bar width = `(value / max) * 100%`.
- **Data Source:** `transactions` prop passed from Dashboard — same filtered set used by the main view. Analytics component is pure/derived (no Firestore calls of its own).
- **Stability Guardrail:** `Analytics.tsx` is fully isolated. No changes to `ExportShare.tsx`, `useProfile.ts`, or Auth logic.
- **Fix #1:** Conflicting peer deps Vite 7 vs PWA plugin solved via `--legacy-peer-deps`.
- **Fix #2:** PDF export failure solved via proper `jspdf-autotable` named import (`import autoTable from 'jspdf-autotable'`).
- **Fix #3:** Missing permissions solved via User-UID based Firestore Rules.
- **Fix #4:** Empty category dropdown — `auth.currentUser` is null at mount; always use `onAuthStateChanged` to wait for auth state before fetching user data.
- **Fix #5:** Duplicate categories — old guard `if (!snapshot.empty) return` was insufficient. Now deduplicates by **name** on every call: removes duplicate docs, then only seeds missing names.
- **Fix #6:** Custom category input — when user selects "Lainnya", reveal a free-text input; submit the custom value, not the "Lainnya" string.
- **Fix #7:** PDF layout — use `(doc as any).lastAutoTable.finalY` for dynamic Y positioning. Always guard with `if (finalY > pageH - 60) doc.addPage()` before drawing post-table elements. Use `didDrawPage` callback for footer on every page. Use `doc.splitTextToSize()` for long disclaimer text.
- **Fix #8:** Hybrid Login & Firestore Mapping — Firebase Auth only supports email login. To support username login: store `username` (lowercase) + `fullName` + `email` in `users/{uid}` Firestore doc on signup. On login, if identifier has no `@`, query `where('username', '==', identifier)` to resolve email first, then call `signInWithEmailAndPassword`. Always sync `fullName` to `auth.displayName` via `updateProfile` so PDF signature and Dashboard greeting use real name, not email.
