If `finalY + POST_TABLE_HEIGHT > pageH - marginBottom`, call `doc.addPage()` before drawing any post-table block. This prevents orphaned signature areas on empty pages. - **Fix #11 ŌĆö Page Numbering (Two-pass):** jsPDF does not know total page count until all pages are rendered. Strategy: render footer with placeholder text, then loop `doc.setPage(i)` after full render, cover placeholder with white rect, and overwrite with final `Halaman X dari Y` string.
- **Fix #12 ŌĆö Signature Alignment:** Signature block (location, line, name) must use `align: 'center'` with `sigX` calculated as the center of the right half of the page: `pageW / 2 + (pageW / 2 - 14) / 2`. Do NOT use a fixed offset like `pageW - 70`.
- **Fix #14 ŌĆö PDF DateTime with Weekday:** Use `weekday: 'long'` in `toLocaleString`/`toLocaleDateString` options to include nama hari (e.g. "Kamis, 9 April 2026"). - **Fix #15 ŌĆö Single Footer with Wrap:** Merge disclaimer + branding into one `FOOTER_FULL` string. Use `doc.splitTextToSize(FOOTER_FULL, pageW - 28)` so text wraps instead of being clipped. Adjust `startY` of footer text upward by `(lines.length - 1) * 3.5` to keep page number at the bottom edge.
- **Fix #16 ŌĆö Table No. Column:** Add auto-numbered `No.` column as index 0 with `cellWidth: 10, halign: 'center'`. Columns `No.`, `Tanggal & Jam`, and `Deskripsi` all use `halign: 'center'`. - **Fix #17 ŌĆö Signature on Last Page Only:** Signature block is drawn only after `lastAutoTable.finalY`, never inside `didDrawPage`. The `POST_TABLE_HEIGHT` guard uses `pageH - MARGIN_BOTTOM - 6` to account for the taller footer area.

## ­¤öÆ DATA ISOLATION STANDARD (PERMANENT RULE)
- **Path-based isolation is the primary mechanism.** All user data lives under `users/{uid}/...`. Never store user financial data at a top-level collection (e.g., `transactions/{id}`).
- **No cross-user query filter needed.** Because isolation is enforced at the Firestore path level, queries within `users/{uid}/transactions` or `users/{uid}/categories` are already scoped ŌĆö no `.where('userId', '==', uid)` required on subcollections.
- **Exception ŌĆö username lookup:** `isUsernameAvailable` and `getEmailByUsername` in `userSetup.ts` query the top-level `users` collection to resolve username ŌåÆ email during login/register. This is intentional and safe ŌĆö only `email` and `username` fields are read, never financial data.
- **Firestore Security Rules (enforced, v2):**
  ```
  match /users/{userId} {
    allow read: if true;  // untuk username lookup saat login/register
    allow write: if request.auth.uid == userId;
    match /transactions/{transactionId} {
      allow read, write: if request.auth.uid == userId;
    }
    match /categories/{categoryId} {
      allow read, write: if request.auth.uid == userId;
    }
  }
  ```
- **Alasan `allow read: if true` pada `/users/{userId}`:** `isUsernameAvailable` dan `getEmailByUsername` perlu query koleksi `users` sebelum user terautentikasi (saat registrasi/login). Hanya field `username` dan `email` yang terbaca ŌĆö tidak ada data finansial di dokumen root user.
- **New features must follow:** Any new collection or document must be placed under `users/{uid}/` and covered by the wildcard rule above. Never create a shared/global collection accessible by multiple users.
- **Design Decision:** Analytics placed in a separate "Analisis" tab (not inline with dashboard) to avoid layout overload on mobile. Tab bar appears below the main header, hidden when Profile view is active.
- **No External Chart Library:** All charts built with native SVG + Tailwind. Zero new dependencies. Keeps bundle size minimal.
- **Charts Implemented:**
  1. **Trend Line Chart** ŌĆö Income vs Expense for last 7 days. SVG polyline with dot markers and date labels. Y-axis normalized to `maxDaily` value.
  2. **Category Pie Chart** ŌĆö Top 5 expense categories. SVG arc paths computed from cumulative angles. Color palette: blue, purple, pink, amber, emerald.
  3. **Monthly Comparison Bar** ŌĆö Side-by-side horizontal bars for this month vs last month (income + expense). Bar width = `(value / max) * 100%`.
- **Data Source:** `transactions` prop passed from Dashboard ŌĆö same filtered set used by the main view. Analytics component is pure/derived (no Firestore calls of its own).
- **Stability Guardrail:** `Analytics.tsx` is fully isolated. No changes to `ExportShare.tsx`, `useProfile.ts`, or Auth logic.
- **Fix #1:** Conflicting peer deps Vite 7 vs PWA plugin solved via `--legacy-peer-deps`.
- **Fix #2:** PDF export failure solved via proper `jspdf-autotable` named import (`import autoTable from 'jspdf-autotable'`).
- **Fix #3:** Missing permissions solved via User-UID based Firestore Rules.
- **Fix #4:** Empty category dropdown ŌĆö `auth.currentUser` is null at mount; always use `onAuthStateChanged` to wait for auth state before fetching user data.
- **Fix #5:** Duplicate categories ŌĆö old guard `if (!snapshot.empty) return` was insufficient. Now deduplicates by **name** on every call: removes duplicate docs, then only seeds missing names.
- **Fix #6:** Custom category input ŌĆö when user selects "Lainnya", reveal a free-text input; submit the custom value, not the "Lainnya" string.
- **Fix #7:** PDF layout ŌĆö use `(doc as any).lastAutoTable.finalY` for dynamic Y positioning. Always guard with `if (finalY > pageH - 60) doc.addPage()` before drawing post-table elements. Use `didDrawPage` callback for footer on every page. Use `doc.splitTextToSize()` for long disclaimer text.
- **Fix #8:** Hybrid Login & Firestore Mapping ŌĆö Firebase Auth only supports email login. To support username login: store `username` (lowercase) + `fullName` + `email` in `users/{uid}` Firestore doc on signup. On login, if identifier has no `@`, query `where('username', '==', identifier)` to resolve email first, then call `signInWithEmailAndPassword`. Always sync `fullName` to `auth.displayName` via `updateProfile` so PDF signature and Dashboard greeting use real name, not email.
